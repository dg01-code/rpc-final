## 一、答辩汇报稿

各位老师好，我是计科223班的丁岗，今天汇报的论文题目是《基于Netty、ZooKeeper与自定义协议的RPC框架设计与实现》，我将从6个方面展开汇报。

### 第一，选题动机

在设计之前，我调研了国内外主流的RPC框架——比如gRPC基于HTTP/2和Protobuf，跨语言能力强但有较重的依赖链；蚂蚁集团的SofaRPC在金融领域表现优异，但源码体量大、定制门槛高。这些框架在专业领域表现卓越，但核心问题在于：**大多数中小规模的分布式应用并不需要如此重的技术栈**，引入一个动辄数十万行源码的框架，无论在部署成本还是二次开发成本上都不占优势。

因此，我从零实现了一个轻量级RPC框架，目标是**可快速上手、可理解每一步实现、可针对业务进行二次开发**。例如在负载均衡方面，我通过策略模式预留了`LoadBalancer`接口，使用者可以实现自己的策略——比如学校的选课系统，在选课高峰期可以优先选择剩余处理能力最强的服务实例。

### 第二，框架整体结构

框架整体分为6层，自底向上分别是：

1. **协议层**：设计了19字节固定头+N字节变长体的二进制协议，包含魔数校验、消息类型、序列化方式、请求ID和数据长度。序列化同样采用策略模式，预留了`Serializer`接口，当前实现了Hessian二进制序列化和JSON序列化两种方式。

2. **注册治理层**：选用ZooKeeper作为注册中心，解决了硬编码服务地址的问题。关键设计是使用临时顺序节点注册服务实例——当服务提供者宕机或会话断开时，ZooKeeper自动删除对应的临时节点，保证消费者不会发现已下线的服务。

3. **传输层**：基于Netty实现，采用主从Reactor线程模型，bossGroup负责接受连接，workerGroup负责IO读写。同时通过CompletableFuture实现了异步转同步，让业务代码以同步语义调用远程服务。

4. **核心调用层**：这是整个框架的灵魂，通过JDK动态代理让用户像调用本地方法一样调用远程服务，封装了服务发现、负载均衡、请求构建、超时重试等全部底层逻辑，实现了调用端与底层实现的解耦。

5. **Spring集成层**：通过两个注解`@RpcService`和`@RpcReference`实现了零配置的服务发布与消费，底层基于Spring的BeanPostProcessor在容器启动时自动扫描、注册服务、注入代理对象。

6. **演示层**：为直观展示框架效果和调用链路，使用HTTP和SSE实现了可视化的调用与追踪系统，还内置了Benchmark工具可进行本地性能测试。

### 第三，自定义协议设计

核心是解决TCP的粘包/半包问题。协议格式为19字节固定头+N字节变长体，解码器继承Netty的`ByteToMessageDecoder`，采用**可重置读指针**的方案：先判断可读字节是否达到19字节，不够直接返回等待；读取头部后，再判断剩余字节是否满足数据长度，不够就调用`resetReaderIndex()`回退读指针等待更多数据到达；数据完整时一次性读取。这比自定义分隔符或固定长度更灵活——消息体大小由头部的长度字段精确控制。

### 第四，异步转同步

Netty本身是异步事件驱动的，但Java业务逻辑通常需要同步等待——下游流程必须拿到上游结果才能继续。我的解决方案是用`CompletableFuture`做桥接：

1. **创建Future**：发送请求前，以requestId为key，将`CompletableFuture<RpcResponse>`存入`ConcurrentHashMap`等待表；
2. **阻塞等待**：调用线程执行`future.get(timeout)`，默认5秒超时；
3. **响应到达**：Netty的IO线程在`channelRead`回调中，根据requestId取出Future，调用`future.complete(response)`；
4. **唤醒返回**：阻塞的调用线程被唤醒，检查异常和类型转换后，将结果返回给业务代码。

整个过程对业务完全透明——用户写`helloService.sayHello("world")`，拿到的就是String，不需要关心Future和回调。

### 第五，ZooKeeper服务注册与发现

一方面避免写死服务地址，实现服务的动态扩缩容；另一方面，利用ZooKeeper临时节点的特性，当服务提供者异常下线时，其对应的临时节点会被ZooKeeper自动删除，消费者下次查询时就不会拿到已下线的地址。节点结构上，服务名为持久节点，实例为临时顺序节点，数据存储的是host:port。

### 第六，Spring集成

通过两个BeanPostProcessor实现：`RpcServiceBeanPostProcessor`在Bean初始化后扫描`@RpcService`注解，提取接口名作为服务名，注册到本地服务表和注册中心；`RpcReferenceBeanPostProcessor`在Bean初始化后扫描`@RpcReference`注解的字段，通过`RpcClientProxy.createProxy()`创建远程代理对象并反射注入，让使用者完全无感知地调用远程服务。

### 问题1：你的自定义协议格式是什么？为什么要这样设计？怎么解决粘包/半包问题？

**协议格式**（`RpcMessage.java:7-8`）：

```
魔数(4B) + 版本(1B) + 消息类型(1B) + 序列化方式(1B) + 请求ID(8B) + 数据长度(4B) + 数据体(NB)
```

总固定头部 = 19字节。魔数值为 `0x44475250`（ASCII "DGPR"），定义在 `RpcMessage.java:13`。

**设计理由**：

| 字段 | 大小 | 作用 |
|------|------|------|
| 魔数 | 4B | 快速识别非法连接/错误数据，防止误读非RPC流量 |
| 版本号 | 1B | 为将来协议升级预留，新老版本可共存 |
| 消息类型 | 1B | 区分REQUEST/RESPONSE，解码后走不同处理分支 |
| 序列化方式 | 1B | 放在协议头中，接收端无需猜测用哪种反序列化器 |
| 请求ID | 8B | long类型，用于请求-响应配对（异步转同步的key） |
| 数据长度 | 4B | 经典Length Field方案，是粘包/半包解法的基础 |

**粘包/半包解决方案**——`RpcMessageDecoder.java:22-43`：

```java
// 第23行：不足19字节，直接return等待更多数据
if (in.readableBytes() < HEAD_LENGTH) {
    return;
}
in.markReaderIndex();          // 第27行：标记读指针
// ... 读取头部 ...
if (in.readableBytes() < dataLength) {  // 第40行：数据体不完整
    in.resetReaderIndex();     // 第41行：回退读指针，等待下一次channelRead
    return;
}
```

这是Netty推荐的**LengthFieldBasedFrameDecoder**理念的简化实现——利用`markReaderIndex()/resetReaderIndex()`实现"不够就退回等、够了就一次读完"，比固定长度更灵活，比分隔符更高效。

### 问题2：为什么选择Netty而不是原生NIO？你的Reactor模型是怎么用的？

**选Netty的理由**：

1. 原生NIO的epoll空轮询bug（JDK未彻底修复），Netty做了检测和重建Selector的兜底；
2. 原生NIO需要自己处理半包/粘包、断连重连、线程分配，Netty通过Pipeline和内置Handler把这些问题封装好了；
3. Netty的ByteBuf做了池化和引用计数，减少了GC压力；
4. 对比原生NIO动辄几百行的模板代码，Netty更精炼。

**主从Reactor模型**（`RpcServer.java:50-51`）：

```java
bossGroup = new NioEventLoopGroup(1);      // 1个线程负责Accept
workerGroup = new NioEventLoopGroup();      // 默认CPU核数线程负责IO读写
```

bossGroup只负责接受新连接（Accept），workerGroup负责已建立连接的读写和业务处理，这是标准的主从Reactor模式。客户端（`RpcClient.java:33`）只用了单个`NioEventLoopGroup`，因为客户端只需要处理一个连接的IO。

### 问题3：异步转同步具体怎么实现的？为什么用CompletableFuture而不是CountDownLatch？

**完整流程**（三段代码协同）：

**步骤1**——发送端创建Future并注册（`RpcClient.java:83-84`）：

```java
CompletableFuture<RpcResponse> future = new CompletableFuture<>();
RpcClientHandler.addPendingRequest(requestId, future);
```

**步骤2**——阻塞等待（`RpcClient.java:96`）：

```java
return future.get(requestTimeoutMs, TimeUnit.MILLISECONDS);
```

**步骤3**——IO线程回调完成（`RpcClientHandler.java:41-45`）：

```java
CompletableFuture<RpcResponse> future = PENDING_REQUESTS.remove(message.getRequestId());
if (future != null) {
    RpcResponse response = SerializerFactory.getSerializer(message.getSerializerType())
            .deserialize(message.getData(), RpcResponse.class);
    future.complete(response);
}
```

**为什么用CompletableFuture而不是CountDownLatch**：

- `CountDownLatch`只能一次性倒计到0，无法传递返回值——还需要一个额外的共享变量来存response；
- `CompletableFuture.complete(value)`既能唤醒阻塞线程，又能直接把结果传过去，一行代码解决两个问题；
- `CompletableFuture`支持超时（`get(timeout)`）、支持异常传播（`completeExceptionally`），而CountDownLatch需要额外处理；
- 业界主流RPC（Dubbo等）也都采用CompletableFuture方案。

**存储结构为什么用ConcurrentHashMap**（`RpcClientHandler.java:20`）：

多线程并发调用时，requestId是`AtomicLong`自增生成的全局唯一ID，`ConcurrentHashMap`保证多线程put/get/remove的原子性。

### 问题4：为什么选ZooKeeper做注册中心？和Nacos、Eureka比有什么优劣？

**选ZooKeeper的理由**：

1. **CP模型**：ZooKeeper基于ZAB协议保证强一致性，服务注册信息不会出现不一致——在RPC场景下，调用到错误地址的代价远大于短暂不可用；
2. **临时节点自动摘除**：`ZookeeperRegistry.java:96-98`用`CreateMode.EPHEMERAL_SEQUENTIAL`创建临时顺序节点，服务宕机后会话超时，节点自动删除，消费者不会再发现已下线的实例；
3. **Curator客户端成熟**：Apache Curator封装了连接管理、重连策略（`ExponentialBackoffRetry`，见`ZookeeperRegistry.java:37`）、session过期重建等，不需要自己处理底层细节。

**对比**：

| 维度 | ZooKeeper | Nacos | Eureka |
|------|-----------|-------|--------|
| 一致性 | CP（强一致） | CP/AP可切换 | AP（最终一致） |
| 临时节点 | 原生支持 | 心跳续约 | 心跳续约 |
| 自身体量 | 较重（需独立集群） | 轻（可嵌入式） | 轻 |
| 适用场景 | 强一致高效 | 配置+注册一体 | Spring Cloud生态 |

本项目是轻量级RPC框架，注册中心的核心需求是"一致性>可用性"——宁可短暂发现不到也不要拿到错误地址，所以选ZooKeeper最合适。

### 问题5：动态代理是怎么实现的？为什么用JDK动态代理而不是CGLIB？

**实现**（`RpcClientProxy.java:50-55`）：

```java
public static <T> T createProxy(Class<T> serviceInterface, ServiceDiscovery serviceDiscovery) {
    return (T) Proxy.newProxyInstance(
            serviceInterface.getClassLoader(),
            new Class<?>[]{serviceInterface, AutoCloseable.class},
            new RpcClientProxy(serviceInterface, serviceDiscovery)
    );
}
```

**invoke方法完整流程**（`RpcClientProxy.java:68-130`）：

1. 过滤`AutoCloseable.close()`和`Object`类方法（本地调用）；
2. 服务发现：`serviceDiscovery.discover(serviceName)` → 获取实例列表；
3. 负载均衡：`loadBalancer.select(serviceName, instances)` → 选一个实例；
4. 构建请求：`new RpcRequest(serviceName, methodName, parameterTypes, args)`；
5. 发送并等待响应（含重试）；
6. 异常传播：如果远程调用抛异常，原样throw出来；
7. JSON模式类型矫正：`coerceJsonNumberResultIfNeeded()`。

**为什么用JDK动态代理**：

- RPC的服务定义本身就是接口——`HelloService`等，JDK代理基于接口，完全够用；
- CGLIB需要额外依赖（cglib-nodep），且通过生成子类字节码实现，在模块化Java（9+）中有访问限制；
- Dubbo默认也是JDK动态代理，只作为可选项提供Javassist/CGLIB。

### 问题6：负载均衡怎么设计的？如果我要加一个加权轮询策略怎么做？

**接口设计**（`LoadBalancer.java:10-12`）：

```java
public interface LoadBalancer {
    ServiceInstance select(String serviceName, List<ServiceInstance> instances);
}
```

策略模式——接口一个方法，两个实现：

- **轮询**（`RoundRobinLoadBalancer.java:13-21`）：`AtomicInteger`自增取模，保证均匀分摊；
- **随机**（`RandomLoadBalancer.java`）：`ThreadLocalRandom`随机选择，无需锁。

**如果要加加权轮询**：只需实现`LoadBalancer`接口即可，例如：

```java
public class WeightedRoundRobinLoadBalancer implements LoadBalancer {
    // 给ServiceInstance加weight字段，按权重分配轮询比例
    @Override
    public ServiceInstance select(String serviceName, List<ServiceInstance> instances) {
        // 加权轮询逻辑...
    }
}
```

然后在`RpcClientProxy`构造时传入自定义实现。**这就是策略模式的好处——符合开闭原则，新增策略不需要修改已有代码。**

### 问题7：ZooKeeper的临时节点怎么用的？如果ZooKeeper挂了会怎样？

**临时节点注册**（`ZookeeperRegistry.java:96-98`）：

```java
String instancePath = client.create()
        .withMode(CreateMode.EPHEMERAL_SEQUENTIAL)   // 临时顺序节点
        .forPath(servicePath + "/instance-", data);   // data = "host:port"
```

**ZK节点结构**：

```
/dg-rpc                                          ← 根路径（持久节点）
  └── com.dg.rpc.demo.api.HelloService           ← 服务名（持久节点）
        ├── instance-0000000001                   ← 实例（临时顺序节点, data="host:port"）
        └── instance-0000000002                   ← 实例（临时顺序节点, data="host:port"）
```

**ZK挂了的影响**：

- **已有连接的调用不受影响**：因为`RpcClient`会缓存已建立的Channel（`channelMap`），不需要每次调用都查ZK；
- **新服务注册/新的服务发现会失败**：`discover()`会抛`RpcException`；
- **改进方向**：可以在客户端加本地缓存，ZK不可用时降级用缓存的服务列表；也可以加ZK集群做高可用——生产环境ZK通常是3-5节点集群。

**Curator的重连策略**（`ZookeeperRegistry.java:37`）：

```java
.retryPolicy(new ExponentialBackoffRetry(1000, 3))
```

初始1秒，最多重试3次，指数退避。ZK短暂闪断后Curator会自动重连。

### 问题8：Spring集成的两个注解是怎么生效的？BeanPostProcessor做了什么？

**@RpcService**——`RpcServiceBeanPostProcessor.java:36-44`：

```java
public Object postProcessAfterInitialization(Object bean, String beanName) {
    if (bean.getClass().isAnnotationPresent(RpcService.class)) {
        Class<?>[] interfaces = bean.getClass().getInterfaces();
        if (interfaces.length > 0) {
            String serviceName = interfaces[0].getName();  // 取第一个接口的全限定名
            serviceMap.put(serviceName, bean);              // 本地注册
        }
    }
    return bean;
}
```

Spring容器刷新后，在`ContextRefreshedEvent`事件中调用`startServer()`，将serviceMap里的服务注册到ZK并启动Netty服务端。

**@RpcReference**——`RpcReferenceBeanPostProcessor.java:23-36`：

```java
public Object postProcessAfterInitialization(Object bean, String beanName) {
    for (Field field : bean.getClass().getDeclaredFields()) {
        if (field.isAnnotationPresent(RpcReference.class)) {
            field.setAccessible(true);
            Object proxy = RpcClientProxy.createProxy(field.getType(), serviceDiscovery);
            field.set(bean, proxy);   // 反射注入代理对象
        }
    }
    return bean;
}
```

**核心原理**：BeanPostProcessor是Spring的扩展点，在每个Bean初始化后都会回调。我们利用它来扫描自定义注解，实现**服务端自动注册、消费端自动注入代理**，和Spring的`@Autowired`机制本质相同。

### 问题9：反射调用那块，如果方法发生异常怎么处理的？

**RequestHandler.java:21-41**：

```java
public RpcResponse handle(RpcRequest request, long requestId) {
    try {
        Object service = serviceMap.get(request.getServiceName());
        Method method = service.getClass().getMethod(methodName, parameterTypes);
        Object result = method.invoke(service, request.getArguments());
        return new RpcResponse(requestId, result);
    } catch (Exception e) {
        return new RpcResponse(requestId, e.getCause() != null ? e.getCause() : e);
    }
}
```

关键处理在第39行：`e.getCause() != null ? e.getCause() : e`。因为反射调用`method.invoke()`抛出的是`InvocationTargetException`，真正的业务异常被包装在cause里。取cause才能在客户端还原出真实的异常类型。

到了客户端（`RpcClientProxy.java:121-123`）：

```java
if (response.hasException()) {
    throw response.getException();   // 直接抛出，让业务代码感知
}
```

实现了**异常传播**——远程调用出错就像本地调用出错一样抛出来。

### 问题10：序列化为什么选Hessian而不是Protobuf？JSON序列化有什么问题？

**选Hessian的原因**：

1. **无需IDL文件**：Protobuf需要写.proto文件再用protoc编译生成Java类，对于轻量级框架太重了；
2. **无需预编译**：Hessian可以直接序列化POJO，零配置；
3. **跨语言**：Hessian本身支持多语言，虽然不如Protobuf高效，但够用；
4. **紧凑度**：二进制格式比JSON紧凑很多，网络传输量小。

**JSON序列化的问题**——在`RpcClientProxy.java:126-160`做了专门的类型矫正：

```java
if (serializerType == SerializerType.JSON) {
    return coerceJsonNumberResultIfNeeded(method.getReturnType(), result);
}
```

因为Gson反序列化时，JSON中的数字默认解析为`Double`——如果方法返回`int add(int a, int b)`，Gson返回的是`Double(3.0)`，必须根据方法返回类型做`intValue()/longValue()`等强转。Hessian序列化保留了Java的类型信息，不需要这步处理。

**策略模式**（`Serializer.java:6-11`）：

```java
public interface Serializer {
    byte[] serialize(Object obj);
    <T> T deserialize(byte[] bytes, Class<T> clazz);
}
```

通过接口+工厂模式（`SerializerFactory`），用户可以扩展任何序列化方式——比如想加Protobuf，只需要实现这个接口。

### 问题11：你的框架和Dubbo有什么本质区别？如果要用于生产还缺什么？

**本质区别**：

- Dubbo是**面向生产**的完整RPC框架，有集群容错（failover/failfast/failsafe/forking）、服务治理（路由/降级/熔断）、注册中心抽象（支持ZK/Nacos/Redis等）、SPI扩展机制、Telnet运维命令等；
- 本框架是**面向学习和轻量场景**的，核心链路完整，但去掉了复杂度——代码量小，全链路可读，可针对性二次开发。

**生产化还缺什么**：

| 缺失能力 | 说明 |
|----------|------|
| 心跳与重连 | 当前`IdleStateHandler`只在空闲时关闭连接，没有自动重连 |
| 连接池 | `channelMap`是最简单的连接复用，缺少最大连接数限制和健康检查 |
| 服务分组/版本号 | 当前只靠接口全限定名做服务key，无法支持同一接口多版本并存 |
| 异步调用 | 当前只支持同步阻塞，生产环境需要支持CompletableFuture异步回调 |
| 流量控制 | 缺少限流/熔断/降级 |
| 注册中心监听 | 当前`discover()`是拉模型，应加ZK的`PathChildrenCache`监听实时推送 |
| 优雅停机 | 当前shutdown只是关EventLoopGroup，没有等待在途请求完成 |

### 问题12：requestId为什么要用AtomicLong？会不会重复？

**代码**（`RpcClient.java:36`）：

```java
private final AtomicLong requestIdGen = new AtomicLong(0);
```

**调用**（`RpcClient.java:75`）：

```java
long requestId = requestIdGen.incrementAndGet();
```

**为什么不重复**：

- `AtomicLong.incrementAndGet()`是原子自增，单JVM内不可能重复；
- requestId只用于当前JVM的`ConcurrentHashMap`做key，不需要跨JVM唯一；
- 如果是分布式全局唯一ID，需要加机器标识或用雪花算法，但RPC的请求配对是在单客户端内完成的，AtomicLong完全足够。

**为什么不用UUID**：

- UUID是36字符的字符串，做Map的key时hashCode计算和equals比较都比long慢；
- long做key的`ConcurrentHashMap`的get/put性能优于String key。

### 问题13：为什么客户端Pipeline里Decoder在Encoder前面？发请求不是先编码吗？

看`RpcClient.java:56-61`：

```java
p.addLast(new IdleStateHandler(0, 0, 30, TimeUnit.SECONDS));
p.addLast(new RpcMessageDecoder());   // 解码器
p.addLast(new RpcMessageEncoder());   // 编码器
p.addLast(new RpcClientHandler());    // 业务处理器
```

**Pipeline是双向的**：

- **入站方向**（读到数据）：`IdleStateHandler → Decoder → Encoder → Handler`——只处理`ByteBuf→RpcMessage`的Decoder和业务Handler会生效，Encoder对入站是透明的；
- **出站方向**（写数据）：`Handler → Encoder → Decoder → IdleStateHandler`——只有Encoder生效，把`RpcMessage`编码成`ByteBuf`写出，Decoder对出站是透明的。

所以Pipeline中Decoder和Encoder的顺序对于各自的方向是不影响的，只要同一个方向上的Handler顺序正确就行。这是Netty的双向Pipeline设计。

### 问题15：假如服务提供者有3个实例，其中一个突然挂了，你的框架会怎样？

分两个阶段看：

**1. ZK临时节点自动摘除**（`ZookeeperRegistry.java:96-98`）：

服务提供者进程挂掉后，ZK的TCP会话断开，session timeout后ZK自动删除临时节点。之后消费者调用`discover()`就不会发现这个实例了。

**2. 摘除之前（时间窗口内）**：

如果恰好在session timeout内调用了已挂的实例，`RpcClient.sendRequest()`会抛异常——要么连接失败，要么`future.get()`超时（5秒）。此时（`RpcClientProxy.java:101-115`）：

```java
for (int i = 0; i < attempts; i++) {
    try {
        response = rpcClient.sendRequest(instance.getHost(), instance.getPort(), request);
        break;
    } catch (Exception e) {
        lastException = e;
        if (i < attempts - 1 && retryIntervalMs > 0) {
            Thread.sleep(retryIntervalMs);
        }
    }
}
```

如果配置了重试（`retries > 0`），会重新走服务发现+负载均衡选下一个实例。

**不足之处**：当前重试是重试同一个实例，没有做"失败后换实例"的逻辑。改进方案是重试时重新调用`loadBalancer.select()`选择下一个实例。
