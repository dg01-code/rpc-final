选题动机

为什么从零手写一个 RPC 框架？
做分布式开发，RPC 是绕不开的技术——服务拆分后，进程内方法调用变成了跨网络远程调用，开发者要处理序列化、网络传输、服务寻址这些底层问题。
现有框架的痛点
Dubbo、gRPC 功能强大，但代码量大、配置多，初学者很难看清"一次RPC调用到底经历了什么"
dg-rpc 的目标
从零手写轻量级RPC框架，把核心链路走通，让每一步都能说清楚为什么这么做
框架定位
不是替代 Dubbo，而是让你能看清 RPC 每一步在做什么——从代理拦截到协议编码，从服务发现到反射调用，链路完整、代码可读
技术选型
Netty 4.1 ZooKeeper 自定义二进制协议 JDK动态代理 Spring Boot 3 SSE可视化
核心价值
看得清：每层代码可直接读懂
讲得明：每个决策都有明确理由
跑得通：端到端链路完整可运行

框架整体架构

六层七模块，按变化点分层
演示层

HTTP桥接 + SSE链路可视化 + Benchmark
Spring集成层
dg-rpc-spring
@RpcService @RpcReference 自动配置
核心调用层
dg-rpc-core
RpcClientProxy · LoadBalancer · Config
传输层
dg-rpc-transport
Netty Client/Server · Handler · Future
注册治理层
dg-rpc-registry
ZooKeeper · InMemory · ServiceDiscovery
协议层
dg-rpc-protocol
RpcMessage · 编解码 · 序列化工厂
按变化点分层——协议格式可能变、注册中心可能换、序列化方式可能换，这些都被隔离成独立模块。改一个不影响其他。
开闭原则
序列化 → 实现 Serializer 接口
注册中心 → 实现 ServiceRegistry + ServiceDiscovery
负载均衡 → 实现 LoadBalancer 接口
模块依赖方向
core 依赖 transport、registry、protocol
transport 不依赖 core，registry 不依赖 core
依赖单向，不循环

自定义协议头设计

19 字节固定头，解决 TCP 粘包/拆包
Magic
4 字节
Version
1 字节
MsgType
1 字节
Serializer
1 字节
RequestId
8 字节
DataLength
4 字节
Magic 0x44475250
"DGRP"的ASCII编码，收到数据先校验魔数，不对直接丢弃——快速排错防误连
RequestId 8字节
AtomicLong递增，异步通信中匹配请求与响应的关键，263 ÷ 100万/秒 ≈ 29万年
DataLength 4字节
配合ByteToMessageDecoder，不够就resetReaderIndex()等待，够了就读取——粘包拆包解法
vs Dubbo 协议头
对比项Dubbodg-rpc
头长度16字节19字节
Magic2字节4字节(更不易碰撞)
序列化标识flag位域(位运算)独立1字节(更直观)
版本字段无有(预留升级)
编解码核心逻辑
Encoder：RpcMessageEncoder extends MessageToByteEncoder
→ 按19字节头顺序写入ByteBuf

Decoder：RpcMessageDecoder extends ByteToMessageDecoder
→ readableBytes < 19 → return 等待
→ markReaderIndex() → 读协议头
→ readableBytes < dataLength → resetReaderIndex() 等待
→ 读取数据体，构造RpcMessage

异步转同步：CompletableFuture 桥接

Netty 是异步的，业务代码期望同步调用
业务代码写 helloService.sayHello("world")，期望拿到 String，不想管 Future 和回调。用 CompletableFuture 做桥接，让异步 Netty 变成同步语义。
1
创建 Future
发请求前，以 requestId 为 key，将 CompletableFuture<RpcResponse> 存入 ConcurrentHashMap 等待表
2
阻塞等待
调用线程执行 future.get(requestTimeoutMs, MILLISECONDS)，默认超时5秒
3
响应到达
RpcClientHandler.channelRead 根据 requestId 从等待表取出 Future，调 future.complete(response)
4
唤醒返回
调用线程被唤醒，检查异常/JSON类型矫正，返回结果给业务代码
两层 Future 的区别
ChannelFuture（Netty层）
→ channel.writeAndFlush() 返回
→ 代表"消息是否写到了网络缓冲区"

CompletableFuture（RPC层）
→ 自己创建和管理
→ 代表"服务端已处理完并返回了结果"

管"发出去了没" vs 管"结果回来了没"
超时与清理
超时 → future.get() 抛 TimeoutException
→ removePendingRequest(requestId) 清理等待表
→ future.cancel(true) 释放资源
→ 包装为 RpcException 抛出

IdleStateHandler：客户端30秒、服务端60秒全空闲超时，自动关闭死连接

ZooKeeper 服务注册与发现

临时顺序节点实现服务自动注册与摘除
服务注册（Provider 启动）
1. 检查持久节点 /dg-rpc/HelloService 是否存在
2. 创建临时顺序节点：/dg-rpc/HelloService/instance-0000000000
3. 节点数据 = host:port
4. 缓存实际路径到 registeredPaths（用于反注册）
服务发现（Consumer 调用）
1. 读取 /dg-rpc/HelloService 下所有子节点
2. 逐一读取节点数据，解析为 ServiceInstance
3. 负载均衡选择一个实例发起调用
当前：每次调用都主动拉取（优化方向见右）
为什么用临时顺序节点？
临时：进程崩了/网络断了，ZK会话超时，节点自动删除
顺序：避免多节点同名冲突，自动加序号后缀
ZooKeeper 树形结构
/dg-rpc ← 根路径(持久) └── HelloService ← 服务节点(持久) ├── instance-0000000000 ← 实例1(临时) data="192.168.1.10:8080" └── instance-0000000001 ← 实例2(临时) data="192.168.1.11:8080"
当前不足与优化方向
问题：每次调用都查 ZK，延迟高、压力大

优化方案：
1. ConcurrentHashMap 本地缓存
2. 5秒过期时间
3. Watcher 监听 NodeChildrenChanged 事件刷新
→ 读取频率从"每次调用"降到"最多5秒一次+事件触发"
ZK 挂了怎么办？
已有连接(channelMap缓存)不受影响
新服务注册不了，新消费者找不到服务
缺少本地缓存降级机制——后续改进方向

Spring 注解集成

两个注解搞定，接入零门槛
@RpcService — 提供者
@RpcService public class HelloServiceImpl implements HelloService { public String sayHello(String name) { return "Hello, " + name + "!"; } }
RpcServiceBeanPostProcessor 在 postProcessAfterInitialization 阶段：
1. 扫描 Bean 上的 @RpcService 注解
2. 取 bean.getClass().getInterfaces()[0].getName() 作为服务名
3. 将 Bean 注册到本地 serviceMap
@RpcReference — 消费者
@Component public class ConsumerBean { @RpcReference private HelloService helloService; }
RpcReferenceBeanPostProcessor 在 postProcessAfterInitialization 阶段：
1. 遍历 Bean 的所有 Field
2. 找到标注 @RpcReference 的字段
3. RpcClientProxy.createProxy() 创建动态代理
4. field.set(bean, proxy) 反射注入
启动顺序控制 — ApplicationListener<ContextRefreshedEvent>
Spring容器启动
→
Bean初始化
→
收集@RpcService
→
ContextRefreshedEvent
→
启动Netty服务端
确保所有 Bean 就绪后才启动服务端，避免"服务已发布但 Bean 未准备好"的竞态问题。@PreDestroy 保证优雅关闭。
RpcAutoConfiguration META-INF/spring/...AutoConfiguration.imports Spring Boot 自动装配

一次完整调用链路

从 sayHello("World") 到拿到结果，端到端全流程
1
业务调用
helloService.sayHello("World") → RpcClientProxy.invoke() 拦截
2
服务发现
ZooKeeper.discover("HelloService") → 获取实例列表 → LoadBalancer.select() 选一个
3
构造 & 编码
RpcRequest → 序列化 → RpcMessage(+19字节协议头) → RpcMessageEncoder → ByteBuf
4
Future 等待
channel.writeAndFlush(msg) → CompletableFuture 放入等待表 → future.get() 阻塞
5
服务端处理
RpcMessageDecoder 校验魔数+拆包 → 反序列化 → RequestHandler 反射执行 method.invoke()
6
响应返回
RpcResponse → 编码 → 网络传输 → RpcClientHandler 按 requestId 匹配 → future.complete() 唤醒
链路可视化：RpcTrace 在每个关键节点投递事件（step 1-6），前端通过 SSE 实时推送，浏览器可看到完整调用路径。
异步转同步
CompletableFuture 等待表
requestId 匹配请求与响应
异常透传
InvocationTargetException → getCause()
→ 封装到 RpcResponse → 客户端 throw
JSON 类型矫正
Gson 反序列化数字默认 Double
代理层按 returnType 矫正为 int/long

JSON 吞吐量约为 Hessian 的 1.76 倍，但这是简单参数场景。Gson 对基础类型解析极快，Hessian 有类型标记、引用表等二进制结构开销。真实环境下复杂对象+网络开销，Hessian 的二进制压缩优势才能体现。框架默认仍为 Hessian。
